HyperIQ Quick Start

Cloudian HyperIQ™ is a solution for monitoring the health and performance of a Cloudian HyperStore system. This document describes how to install HyperIQ, connect it to your HyperStore system, and start using it.

Requirements

Requirement Description
Virtualization or
containerization platform

HyperIQ is available in two forms, supporting different types of operating platforms:

  • A virtual appliance, for installation on VMware (ESXi 5.5 or newer, or Workstation 10.x or newer) or VirtualBox.
  • A containerized application, for installation on Docker (requires docker 19.03 or newer and docker-compose 1.17 or newer; Kubernetes is not currently supported)
HyperIQ OVA or installer

From the Cloudian Support portal, download either:

  • hyperiq-1.1.0.ova, for installing HyperIQ as a virtual appliance
  • hyperiq-1.1.0.bin or hyperiq-1.1.0.app or hyperiq-1.1.0.exe, for installing HyperIQ in a Docker environment on Linux or MacOS or Windows, respectively
HyperStore system

HyperIQ has been verified to work with HyperStore 7.1 and newer.

A single HyperIQ instance can serve a full HyperStore system, even if the HyperStore system spans multiple data centers and/or multiple service regions.

To set up HyperIQ you will need the IP address of one of your HyperStore nodes -- it can be any node -- and the root user password for SSH access to the node (or the sa_admin user password if you use the HyperStore Shell to manage your HyperStore nodes). Alternatively, for root user access you can use an SSH key rather than a password.
HyperIQ access to
HyperStore

HyperIQ must be able to access the following ports on every HyperStore node in your system, in all of your HyperStore data centers and service regions:

  • 22 (only during HyperIQ set-up)
  • 9999 (for ongoing collection of HyperStore health and performance metrics)

If you are using a third party firewall to restrict access to HyperStore, you must configure the firewall to allow HyperIQ to access these ports on each HyperStore node.

If you are running HyperStore 7.2.0 or 7.2.1 and are using the built-in HyperStore Firewall, you must disable the HyperStore Firewall if you want to use HyperIQ. The HyperStore Firewall in HyperStore 7.2.0 and 7.2.1 is incompatible with using HyperIQ. You can re-enable the HyperStore Firewall when you upgrade to HyperStore 7.2.2 (in this version the HyperStore Firewall does support HyperIQ). For instructions on disabling the HyperStore Firewall, in your HyperStore 7.2.x documentation see section 4.3.2 "HyperStore Firewall".

Do not disable the HyperStore Firewall if your HyperStore 7.2.0 or 7.2.1 deployment lacks a dedicated internal network for HyperStore nodes (for context see Knowledge Base article # 000002428 on the Cloudian Support Portal). Instead, wait until you've upgraded to HyperStore 7.2.2 before using HyperIQ.

Installing HyperIQ as a Virtual Appliance

This section describes how to install HyperIQ version 1.1 as a virtual appliance. For installing HyperIQ in an existing Docker environment, jump ahead to Installing HyperIQ in a Docker Environment.

Upgrade to HyperIQ 1.1 from the HyperIQ 1.0 virtual appliance is not supported. If you have been using HyperIQ 1.0, you can do a fresh installation of HyperIQ 1.1 as a separate VM, and then continue to use HyperIQ 1.0 for a while until your HyperIQ 1.1 application has accumulated enough HyperStore monitoring data to meet your needs.

Deploying the HyperIQ VM

Use your virtualization platform console to deploy a new virtual machine (VM) from the HyperIQ OVA file.

For example, for VMware ESXi:

  1. From the device on which you downloaded the HyperIQ OVA file, connect to your ESXi Server console and log in.
  1. On the left side of the ESXi console, select Virtual Machines.
  2. Click Create / Register VM. This launches the New virtual machine creation wizard.
  3. For the VM creation type select “Deploy a virtual machine from an OVF or OVA file”, then click Next.
  4. Enter your desired name for the virtual machine, then click in the file selection area of the wizard screen and browse to the HyperIQ OVA file that you downloaded. After selecting the OVA file, click Next.
  5. In the remaining screens of the VM creation wizard, accept the default values (or make adjustments if you wish).
  6. In the wizard's Ready to complete screen, review the attributes of the new VM and then click Finish.

Although the wizard will close and the new VM will appear in Virtual machines list within seconds, it will take some time -- perhaps 10-15 minutes depending on your environment -- for creation of the VM to complete. You can track the progress by opening the Recent tasks panel at the bottom of the Virtual machines list.

The HyperIQ VM that is created has these specs:

  • 2 CPU cores, 8GB RAM, one 150GB disk for the OS and data
  • OS = Ubuntu Linux 64-bit
  • hostname = hyperiq

The mature open source technologies Prometheus and Grafana, which HyperIQ leverages for the collection, analysis, and visualization of HyperStore monitoring data, are pre-installed on the HyperIQ VM.

The HyperIQ VM stores its data locally, on the 150GB disk, without any replication. In a production environment you should take appropriate precautions to improve data durability, such as performing regular backups of the VM.

Integrating the HyperIQ VM with HyperStore

If the HyperIQ VM is not already powered on, power it on now. Then access the VM's console, logging in with user name cloudian and password Cloudian1!

For example, for VMware ESXi:

  1. In the Virtual machines list, click the newly created VM.
  2. In the VM's administration screen, click Console, then from the drop-down list choose where you want the console to display (such as within the same browser tab or in a different tab).
  3. Log in with user name cloudian and password Cloudian1!.

When prompted, create a new password for the 'cloudian' user. (The change password prompt will require you first to enter again your "current UNIX password", which is Cloudian1!).

Once you're logged in, a HyperIQ installer tool is available to lead you through the process of integrating HyperIQ with your HyperStore cluster. To launch the installer, enter this command:

$ iqinstall

By default during the installation process you will need to supply a password so that HyperIQ can access HyperStore via SSH as the root user (or as the sa_admin user if you have configured your HyperStore system to use the HyperStore Shell). If instead of a password you want to use an SSH key that you have already installed on the HyperIQ host, launch the installer as iqinstall -k <ssh key filename>. The option of using an SSH key rather than a password is supported for root user access only. For accessing HyperStore as the sa_admin user, only password authentication is supported currently.

When you launch the installer you will be first prompted for configure basic networking settings for the HyperIQ node, such as hostname or FQDN, IP address, networking mode (DHCP or static), and so on. At some prompts a default value is displayed, which if you wish you can accept by pressing Enter.

Once the installer applies the network settings, it completes the installation of HyperIQ packages and then prompts you to provide the information needed to integrate HyperIQ with your HyperStore system. When prompted by the installer, provide the IP address of one of your HyperStore nodes (it can be any node, and the IP address must be reachable by HyperIQ), and the user as which to connect to HyperStore (root or sa_admin), and that user's password. Note that the password prompt will not appear if you're using an SSH key instead. From the one HyperStore IP address that you provide, the HyperIQ installer will automatically obtain the IP addresses of the rest of your HyperStore nodes. The installer then installs "exporters" on each HyperStore node (the exporters make HyperStore monitoring statistics available to HyperIQ). If your HyperStore system spans multiple data centers and/or multiple service regions, the HyperIQ installer installs exporters on all of your HyperStore nodes in all data centers and service regions. As its final step the installer configures Prometheus and Grafana on the HyperIQ VM.

After the installer completes and returns you to the console prompt, you can exit the console.

For information about managing an HyperIQ node, see Managing a HyperIQ Node

For its access to port 9999 on your HyperStore nodes, HyperIQ uses HTTPS (TLS) with Basic Authentication. The HyperIQ setup tool automatically configures these security mechanisms on HyperIQ and the HyperStore nodes, using a self-signed certificate for TLS and a randomly generated password for Basic Authentication.

Additional Steps If Your HyperStore Cluster Uses an Inaccessible Private Network

If your HyperStore nodes are configured to use a private network for intra-cluster communication and your HyperIQ node is not attached to that private network, extra steps are required to successfully integrate HyperIQ with your HyperStore cluster:

  1. As described in the preceding section, during the initial HyperIQ installation run you should provide a reachable IP address for one HyperStore node (which would be the node's public IP address, if HyperIQ is not on the HyperStore nodes' private network). Using this reachable address, HyperIQ retrieves from that HyperStore node the HyperStore "service map" (cloudianservicemap.json). However, because the HyperStore node IP addresses in that service map are the private network IP addresses, HyperIQ is unable to connect to the other HyperStore nodes and therefore the setup fails and shows connection errors.
  2. After the initial setup run fails, on the HyperIQ node change into the directory $HOME/.config/cloudian/hyperiq and manually edit the cloudianservicemap.json file to replace each HyperStore node's private IP address with its public IP address.
  3. Run the installation again, and when prompted with 'Refresh cloudianservicemap.json?' answer no.

This time the installation should succeed.

If you subsequently add new nodes to your HyperStore system, and are updating your HyperIQ configuration by using iqmgr setup (see Managing a HyperIQ Node), you will need to go through a similar process of first having the setup fail (as HyperIQ retrieves the updated service map file that includes the new HyperStore nodes); and then manually editing HyperIQ's copy of the service map file to replace private IP addresses with the corresponding public IP addresses; and then running iqmgr setup again.

Installing HyperIQ in a Docker Environment

To install HyperIQ version 1.1 in an existing Docker environment:

  1. On the host machine on which you want to install HyperIQ, log in as a user who has permissions to run docker commands. In a working directory on the host machine, download or copy the HyperIQ installer appropriate for the host operating system (hyperiq-1.1.0.bin for Linux or hyperiq-1.1.0.app for MacOS or hyperiq-1.1.0.exe for Windows).

You must use a command terminal for the remainder of this procedure.

  1. Change into the working directory in which you've put the HyperIQ installer file.
  2. For Linux or MacOS, run the following command to make the installer file executable (for Windows, skip ahead to Step 4).

Linux:

chmod +x hyperiq-1.1.0.bin

MacOS:

chmod +x hyperiq-1.1.0.app
  1. Start the installation.

Linux:

./hyperiq-1.1.0.bin install

MacOS:

./hyperiq-1.1.0.app install

Windows:

hyperiq-1.1.0.exe install

By default during the setup process you will need to supply a password so that HyperIQ can access HyperStore via SSH as the root user (or as the sa_admin user if you have configured your HyperStore system to use the HyperStore Shell). If instead of a password you want to use an SSH key that you have already installed on the HyperIQ host, append -k <ssh key name> to the install command (for example ./hyperiq-1.1.0.bin install -k <ssh key filename>). The option of using an SSH key rather than a password is supported for root user access only. For accessing HyperStore as the sa_admin user, only password authentication is supported currently.

  1. When prompted by the installer, provide the IP address of one of your HyperStore nodes (it can be any node, and the IP address must be reachable by HyperIQ), and the user as which to connect to HyperStore (root or sa_admin), and that user's password. Note that the password prompt will not appear if you're using an SSH key instead.

The installer:

  • Creates three containers: one each for the mature open source technologies Prometheus and Grafana (which HyperIQ leverages for the collection, analysis, and visualization of HyperStore monitoring data), and one for HyperIQ management utilities.
  • Installs "exporters" on each of your HyperStore nodes. The exporters make HyperStore monitoring statistics available to HyperIQ. If your HyperStore system spans multiple data centers and/or multiple service regions, the HyperIQ installer will install exporters on all of your HyperStore nodes in all data centers and service regions.
  • Configures Prometheus and Grafana.

For special instructions that apply when your HyperStore nodes use a private network to which HyperIQ is not attached, see Additional Steps If Your HyperStore Cluster Uses an Inaccessible Private Network. Note that the location of the service map file described in those special instructions depends on the host operating system:
* Linux: $HOME/.config/cloudian/hyperiq
* MacOS: $HOME/Library/Application\ Support/cloudian/hyperiq
* Windows: %USER%\AppData\Roaming\cloudian\hyperiq

For information about managing an HyperIQ node, see Managing a HyperIQ Node

For its access to port 9999 on your HyperStore nodes, HyperIQ uses HTTPS (TLS) with Basic Authentication. The HyperIQ installer automatically configures these security mechanisms on HyperIQ and the HyperStore nodes, using a self-signed certificate for TLS and a randomly generated password for Basic Authentication.

Getting Started with HyperIQ

To connect to the HyperIQ UI point a browser to the following URL:

http://<HyperIQ host name or IP address>:3000

Log in with username cloudian_admin and password admin. You will then be prompted to change the password for the cloudian_admin user.

Viewing HyperIQ Dashboards and Panels

When you've logged in to HyperIQ for the first time you are at the HyperIQ Main dashboard. This is one of five HyperIQ dashboards:

  • HyperIQ Main -- HyperStore host OS level metrics such as storage capacity consumption and availability, CPU and memory utilization, and network traffic. This dashboard also displays HyperStore system health alerts.
  • Hardware Status -- For HyperStore appliances, Intelligent Platform Management Interface (IPMI) sensor data such as hardware component temperatures, fan speeds, and power usage.
  • Services -- HyperStore service level metrics such as up/down status and resource consumption for each service that runs on HyperStore hosts, such as the S3 Service, the Admin Service, and Cassandra.
  • Cassandra Status -- Cassandra database metrics such as those pertaining to space consumption, compaction, and performance.
  • S3 Analytics -- HyperStore S3 metrics regarding users, buckets, and transactions.

The S3 Analytics dashboard is available only if your HyperStore system license includes support for the Enterprise version of HyperIQ. For information about acquiring a HyperStore license that supports the Enterprise version of HyperIQ, contact Cloudian Support.

To switch between dashboards, in the upper left of the UI click the dashboard title button.

A list of the available dashboards then appears, and you can click on a dashboard name to go to that dashboard.

Each dashboard consists of a set of "panels", with each panel providing a graphical representation of one metric (such as CPU Utilization or Top S3 Operations). In each dashboard, panels are grouped into panel folders. When you first access a dashboard, the first panel folder is open so that the panels from that folder display. To see the full list of panel folders available in a dashboard, collapse the open folder.

Filtering the Data in a Dashboard

For each dashboard, in the top left of the HyperIQ UI you can filter the data in ways relevant to that dashboard. For example in the HyperIQ Main dashboard you can filter data by service region, data center, and node.

As a second example, in the S3 Analytics dashboard you can filter data by user and bucket.

If you filter dashboard data using the filters top left of the HyperIQ UI, your chosen filter applies to all panels in the dashboard.

HyperIQ currently lacks the ability to partition HyperStore S3 user data by user group. One consequence of this is that in the unlikely event that two users in different groups have the same HyperStore user name, the S3 Analytics dashboard will treat their activity as if it was generated by a single user.

Changing the Time Range for a Dashboard

In the top right of the HyperIQ UI, you can change the time period for which a dashboard's panels display historical data. By default, such panels display data for the past hour. If you change the time period the change applies to all time period sensitive panels on the dashboard.

You cannot view data that's previous to the date of your HyperIQ installation. Also, the current version of HyperIQ does not support viewing data that's more than 30 days old.

Sorting the Data in a Panel

In panels that display data in columns, you can sort the data by clicking on column headings.

Viewing a Larger Version of a Panel

To enlarge a panel, click the panel title and then click View.

Viewing Information About the Data in a Panel

For brief information about the data in a panel, hold your cursor over the "i" to the left of the panel name.

For More Information About Working with Dashboards and Panels

HyperIQ is built on Grafana data visualization technology and exposes much of the Grafana native functionality. For more information about Grafana capabilities see the Grafana official documentation or community resources, which are available on the web.

Adding HyperIQ Users

The cloudian_admin user is a "Super Admin" user with full HyperIQ permissions, and is the only user who can add new HyperIQ users. When you are logged in as the cloudian_admin user you can create new HyperIQ users and then assign each user one of three roles with differing permission levels:

  • Admin: Can view, edit, and create dashboards and panels; configure alerts and notifications; and manage existing users (such as by changing a user's role or deleting a user). Cannot add new users.
  • Editor: Can view, edit, and create dashboards and panels; and configure alerts and notifications. Cannot add or manage users.
  • Viewer: Can view dashboards and panels. Cannot edit or create dashboards and panels, configure alerts or notifications, or add or manage users.

In addition to the cloudian_admin user with the Super Admin role, HyperIQ includes one pre-configured user with the Editor role. The user name is org_editor and the password for this user's first login to the HyperIQ UI is admin. Upon this user's first login they will be prompted to change their password.

Adding a HyperIQ User

  1. On the left side of the HyperIQ UI, click the Server Admin icon.

  1. In the Server Admin page, click New user.
  2. Enter the information for the new user and then click Create.

By default all new users are created with the Viewer role. If you want a user to have more than view permissions you can explicitly assign them a role as described below.

Assigning a HyperIQ User a Role

  1. On the left side of the HyperIQ UI, click the Configuration icon.

  1. In the Configuration page, select the Users tab.
  2. In the list of users that displays, use the Role column to change a user's assigned role.

The Configuration page's Users tab also shows an Invite button. The Invite function -- for sending email invitations that enable recipients to become HyperIQ users if they wish to -- is not supported in the current version of HyperIQ.

Assigning Permissions for a Specific Dashboard

Depending on your HyperIQ use case, simply assigning users one of the three roles (Admin, Editor, or Viewer) may be sufficient for managing HyperIQ user permissions. However, if you wish you can implement more granular permissions, at the dashboard level.

A good example would be if you want users with the Viewer role to be able to view only the S3 Analytics dashboard and not the HyperIQ Main dashboard or the Services dashboard. To do so:

  1. Go to the HyperIQ Main dashboard, and then in the upper right of the UI click the Dashboard Settings icon:

  1. In the HyperIQ Main / Settings page select Permissions.

  1. In the Viewer role row, click the red X at the right to remove the Viewer role from the list of roles that have permissions for this dashboard.
  2. On the left side of the HyperIQ Main / Settings page click Save (optionally you will be able to enter a comment about the saved change).
  3. Repeat Steps 1 through 4 for the Services dashboard.

As a variation of the example above, you could set things up so that most users with the Viewer role can view only the S3 Analytics dashboard but certain chosen users could also have view-only permissions for the HyperIQ Main and Settings dashboards. To do this, follow the steps above, but for both the HyperIQ Main dashboard and the Services dashboard, after removing the Viewer role from the permissions list click Add Permission. Then select the user to whom to grant permission, select View as the permission level, and click Save.

If you plan to make extensive use of dashboard level permissions, you may wish to organize your users into teams, and then after doing so you can set a dashboard level permission for a whole team. To organize users into teams, on the left of the HyperIQ UI click the Configuration icon and then in the Configuration page select the Team tab.

When a dashboard's permissions are configured such that for some users there is a discrepancy between their role-based permission, a team-based permission, and a user-specific permission, the highest degree of permission applicable to the user takes precedence. For example if a user with the Viewer role belongs to a team that has Edit permissions to the dashboard, then the user has Edit permissions for the dashboard.

Setting Up Alerts and Notifications

The HyperIQ Main dashboard includes an Active Alerts panel that shows the list of currently configured alerts and the status of each alert (OK or Alerting).

HyperIQ includes pre-configured alerts for:

  • Node down
  • Service down
  • Disk failure
  • A node's CPU utilization is over 80%
  • A node's RAM utilization is over 80%
  • A node's storage capacity utilization is over 80%
  • A node's transmit or receive network traffic is over 1Gb/sec

If you wish you can edit the configuration of these alerts as described in Editing Alerts.

Each alert is based on data in a particular panel from a dashboard. Panels for which an alert has been configured are indicated by a heart-shaped icon next to the panel title (a green heart indicates an OK status for the alert and a red heart indicates an Alerting status).

By default, there is no sending of notifications in connections with alerts -- rather, the alerts appear only in the HyperIQ UI. You can enable notifications as described in Enabling Email Notifications.

Editing Alerts

To edit an existing alert, in the HyperIQ Main dashboard's Alert List panel, click on the alert that you want to edit. (Alternatively, on the left side of the HyperIQ UI you can click the bell-shaped Alerting icon and then in the Alert Rules list, click on the alert rule you want to edit). This opens a page for editing the rule, where you can for example change the alerting threshold for a threshold-based alert such as High CPU Utilization.

After making any change to an alert rule, at the top right of the HyperIQ UI click the Save Dashboard icon to save your change.

For more detail on configuring alert rule conditions, see the Grafana documentation available on the web.

Enabling Email Notifications

To enable email notifications for HyperIQ alerts you must:

  1. Configure HyperIQ integration with your existing SMTP server by editing a Grafana configuration file on the HyperIQ node.
  2. Use the HyperIQ UI to configure one or more "notification channels" (with each channel being a named set of email addresses).
  3. Use the HyperIQ UI to apply the notification channel(s) to all alert rules or selected alert rules.

To configure HyperIQ integration with an SMTP server, log into the HyperIQ VM -- not the UI but rather the node itself, either by way of your virtualization platform console or an SSH client -- with Unix user name cloudian and the password that you created when you installed and set up the HyperIQ node. Use a text editor to open the configuration file /etc/grafana/grafana.ini. Then edit the "SMTP / Emailing" section of the file -- the default settings are shown below -- so that enabled = true and the rest of the settings match your SMTP environment (including the SMTP server host name or IP address) and your notification email preferences.

#################################### SMTP / Emailing ##########################
[smtp]
;enabled = false
;host = localhost:25
;user =
# If the password contains # or ; you have to wrap it with triple quotes. Ex """#password;"""
;password =
;cert_file =
;key_file =
;skip_verify = false
;from_address = admin@grafana.localhost
;from_name = Grafana
# EHLO identity in SMTP dialog (defaults to instance_name)
;ehlo_identity = dashboard.example.com

[emails]
;welcome_email_on_sign_up = false

After saving your edits and closing the configuration file, restart the Grafana service to apply your configuration changes:

$ sudo systemctl restart grafana-server.service

Next, use your browser to connect to the HyperIQ UI (http://<IP address of HyperIQ VM>:3000) and log in. Then on the left of the UI click the Alerting icon.

In the Alerting page, select the Notification Channels tab, then click Add Chanel. This displays an interface for configuring a new notification channel.

Enter a name for the channel (for example "System Admins"), select whether or not you want the channel to be applied to all HyperIQ alerts by default (so that all alerts result in emails being sent to this channel), and enter one or more email addresses for this channel (for example the email address[es] of the administrator[s] of your Cloudian systems). Then click Save.

You can create additional notification channels in the same way if you wish.

To send a test email to one of your saved channels, in the Notification Channels list click on the channel name, and then click Send Test.

If you did not apply your notification channel(s) to all alerts by default, you can instead apply the channel(s) only to specific alert rules of your choosing. In this way you can have some alert rules result in email notifications while others do not. To do so, in the HyperIQ Main dashboard's Alert List, click on an alert for which you want to enable notifications. In the alert edit page, next to Send To, click the "+" sign.

This displays a drop-down list of your notification channels from which you can select. You can also edit the notification message if you wish. When you're done, at the top right of the HyperIQ UI click the Save Dashboard icon to save your change.

Managing a HyperIQ Node

HyperIQ includes a management tool named iqmgr. The directory location of this tool depends on your operating system type:

Operating System Location of iqmgr Tool (and Other HyperIQ Files)
Linux (including the HyperIQ virtual appliance) $HOME/.config/cloudian/hyperiq
MacOS $HOME/Library/Application\ Support/cloudian/hyperiq
Windows %USER%\AppData\Roaming\cloudian\hyperiq

If you installed HyperIQ as a virtual appliance, you can run iqmgr from any directory. If you installed HyperIQ into an existing Docker environment, you must change into the applicable iqmgr directory from the table above to run iqmgr, or else include the directory path when running iqmgr (or for convenience you could, for example, create a symlink to the tool).

With the iqmgr tool you can perform the following HyperIQ management tasks:

Management Task iqmgr Command

Reconfigure integration with HyperStore

Do this if you change your HyperStore cluster composition such as by adding a node.

iqmgr setup

Note: When prompted for whether you want to refresh the local cloudianservicemap.json file, answer yes (unless you have HyperStore on a private network to which HyperIQ is not attached and you've manually edited HyperIQ's copy of the service map file as described in the HyperIQ setup procedure).
Display HyperIQ container logs iqmgr logs

Note: Additional log files -- for installer output and iqmgr output -- are located in the same directory as iqmgr.
Check or change HyperIQ virtual appliance network settings (applicable only to the virtual appliance and not to HyperIQ installed in an existing Docker environment)
  • iqmgr network status
  • sudo iqmgr network config
Display the HyperIQ version number iqmgr version
Display iqmgr command help igmgr <command> --help

Even if you install HyperIQ as a virtual appliance (using the OVA file), HyperIQ is implemented as a containerized application that uses Docker software that's packaged with the appliance. Therefore even on the HyperIQ virtual appliance you can use docker and docker-compose commands, such as docker ps to list the currently running containers.

Uninstalling HyperIQ

To uninstall HyperIQ after having installed it to an existing Docker environment, run hyperiq-1.1.0.bin uninstall or hyperiq-1.1.0.app uninstall or hyperiq-1.1.0.exe uninstall, for uninstalling on Linux or MacOS or Windows, respectively.

This action will stop and remove Docker containers, images, networks and volumes, and the logs and tools directory. Your HyperIQ data will be lost (you will lose accumulated HyperStore monitoring data stored in the HyperIQ database, from the period during which you've had HyperIQ running). If instead you want to reinstall the HyperIQ application without losing your existing data, then instead of uninstalling you can simply run hyperiq-1.1.0.<ext> install again. This will reinstall the HyperIQ containers without loss of existing HyperIQ data.

To uninstall HyperIQ after having installed it from the OVA, you can simply delete the VM using your virtualization platform management UI. Or to reinstall the HyperIQ application without losing existing HyperIQ data, log into the VM as the cloudian user and run iqinstall again.
